# Plan 88-02: Admin Route Security Audit

## Objective

Audit every route under /admin/* to verify it is wrapped in SuperAdminRoute and inaccessible to non-super_admin users. Document findings in the audit report.

## Tasks

### Task 1: Audit Router-Level Protection

<read_first>
- src/platform/router/AppRouter.tsx (all route definitions — admin routes at lines 73-89)
- src/platform/auth/SuperAdminRoute.tsx (the guard component implementation)
- src/platform/auth/ProtectedRoute.tsx (standard auth guard for comparison)
</read_first>

<action>
Verify the following in `src/platform/router/AppRouter.tsx`:

1. **All admin routes are nested under SuperAdminRoute wrapper.** The structure must be:
   ```
   <Route element={<SuperAdminRoute>...<AdminLayout />...</SuperAdminRoute>}>
     <Route path="/admin" ... />
     <Route path="/admin/users" ... />
     <Route path="/admin/tenants" ... />
     <Route path="/admin/tenants/:orgId" ... />
     <Route path="/admin/modules" ... />
     <Route path="/admin/connectors" ... />
     <Route path="/admin/product-docs" ... />
     <Route path="/admin/settings" ... />
   </Route>
   ```

2. **No admin route exists outside the SuperAdminRoute wrapper.** Search the entire Routes block for any `/admin` path that is NOT inside the SuperAdminRoute parent.

3. **SuperAdminRoute checks both authentication and authorization:**
   - `isSignedIn` check (redirects to sign-in if not authenticated)
   - `user.publicMetadata.super_admin === true` check (redirects to "/" if not super_admin)
   - Loading state shows generic "Carregando..." text, NOT admin content

4. **Enumerate all admin routes found** and their protection status (wrapped/unwrapped).

Expected admin routes (8 total):
- `/admin` — AdminDashboard
- `/admin/users` — UsersPage
- `/admin/tenants` — TenantsPage
- `/admin/tenants/:orgId` — TenantDetailPage
- `/admin/modules` — ModulesPanel
- `/admin/connectors` — ConnectorsPanel
- `/admin/product-docs` — ProductDocsPage
- `/admin/settings` — SettingsPanel

If ANY route is missing SuperAdminRoute protection: wrap it in SuperAdminRoute or move it inside the existing wrapper.
</action>

<acceptance_criteria>
- All 8 admin routes in AppRouter.tsx are nested inside the SuperAdminRoute wrapper element
- No `/admin` path exists in AppRouter.tsx outside the SuperAdminRoute parent Route
- SuperAdminRoute.tsx contains `user?.publicMetadata?.super_admin !== true` check that returns `<Navigate to="/" replace />`
- SuperAdminRoute.tsx contains `!isSignedIn` check that returns `<RedirectToSignIn />`
- SuperAdminRoute.tsx loading state does NOT render any admin-specific content (only generic "Carregando...")
</acceptance_criteria>

<action>
For each admin page component:

1. Verify it is ONLY imported in `src/platform/router/AppRouter.tsx` — no other file imports it
2. Verify it has no named exports that could be used to render admin content outside the protected route
3. Verify AdminLayout and AdminSidebar are only used within the SuperAdminRoute wrapper

Search commands:
```bash

grep -r "from.*pages/admin" src/ --include="*.tsx" --include="*.ts"
```

If any admin component is imported outside AppRouter.tsx (except for admin components importing sub-components from within admin/): document the finding and assess if it leaks admin UI to non-super_admin users.
</action>

<acceptance_criteria>
- Admin page components (AdminDashboard, TenantsPage, TenantDetailPage, UsersPage, ModulesPanel, ConnectorsPanel, SettingsPanel, ProductDocsPage) are imported ONLY in AppRouter.tsx

</acceptance_criteria>

### Task 3: Verify Edge Function Auth

<read_first>
- supabase/functions/admin-users/index.ts
- supabase/functions/admin-tenants/index.ts
</read_first>

<action>
Verify that each admin edge function enforces super_admin at the API level:

1. `admin-users/index.ts` must contain super_admin JWT claim validation before returning any data
2. `admin-tenants/index.ts` must contain super_admin JWT claim validation before returning any data

Check for:
- JWT token extraction from Authorization header
- `payload.super_admin` check (both boolean `true` and string `"true"`)
- 403 Forbidden response when check fails
- CORS headers allowing only the expected origins

Document each function's auth pattern in the audit report.
</action>

<acceptance_criteria>
- `supabase/functions/admin-users/index.ts` contains `payload.super_admin` check that returns 403 on failure
- `supabase/functions/admin-tenants/index.ts` contains `payload.super_admin` check that returns 403 on failure
- Both functions extract JWT from Authorization header before processing requests
</acceptance_criteria>
