Create `supabase/migrations/012_scope_data_migration.sql` ŌĆö a data migration that sets `scope` correctly on all existing documents in the database.

**Rules for scoping:**
- `scope = 'product'`: docs whose `slug` starts with `sdk/` ŌĆö these are SDK/onboarding docs that should be visible to all tenants
- `scope = 'tenant'`: everything else (processo/, ferramentas/, padroes/, referencias/) ŌĆö these are FXL enterprise docs, already tenant-isolated by `org_id`

Since migration 011 already set all rows to `scope = 'tenant'` as the default, this migration only needs to UPDATE the `scope = 'product'` rows.

```sql
-- Migration 012: Backfill scope for existing documents
-- Product docs: SDK/onboarding content visible to all tenants
-- Tenant docs: FXL process/tooling content ŌĆö already defaulted to 'tenant' by migration 011

BEGIN;

-- Set SDK docs to product scope
UPDATE documents
SET scope = 'product'
WHERE slug LIKE 'sdk/%';

-- Verify: log counts for audit (advisory, not blocking)
DO $$
DECLARE
  product_count INT;
  tenant_count INT;
BEGIN
  SELECT COUNT(*) INTO product_count FROM documents WHERE scope = 'product';
  SELECT COUNT(*) INTO tenant_count FROM documents WHERE scope = 'tenant';
  RAISE NOTICE 'After migration 012: % product docs, % tenant docs', product_count, tenant_count;
END $$;

COMMIT;
```

If there are currently no docs with `slug LIKE 'sdk/%'` in the database (because SDK docs were not yet added), this migration is a no-op and that is correct ŌĆö product docs will be added by the super admin via the new `/admin/product-docs` page.

Note on `docs/` filesystem sync: The `.md` files in `docs/` under `processo/`, `ferramentas/`, `padroes/` are the FXL process docs. When `make sync-up` is run, those are upserted into the `documents` table. Their `scope` defaults to `'tenant'` per migration 011's column default, so no change is needed in the sync script. Product docs created via the admin UI are never synced back to the filesystem ŌĆö they live only in Supabase.
