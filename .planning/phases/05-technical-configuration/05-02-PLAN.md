  <action>
Create `tools/wireframe-builder/lib/config-resolver.ts` implementing the resolveConfig pure function per research Pattern 2.

**resolveConfig(blueprint, technical, branding?) -> GenerationManifest:**

1. **Validate slug match**: `if (blueprint.slug !== technical.slug) throw new Error(...)`.

2. **Resolve branding**: Use `resolveBranding(branding)` from existing branding.ts. Import via `@tools/wireframe-builder/lib/branding`.

3. **Derive Supabase schema** via internal `deriveSupabaseSchema(technical)`:
   - For each `reportType`: Create a table with columns derived from column mappings (sourceColumn -> snake_case column name, dataType -> Postgres type mapping: text->TEXT, number->NUMERIC, date->DATE, currency->NUMERIC(15,2)). Add id UUID PK, period_month INT, period_year INT, created_at TIMESTAMPTZ, uploaded_by TEXT columns.
   - For each `settingsTable`: Create a lookup table with id UUID PK, columns from settings.columns (all TEXT), is_active BOOLEAN DEFAULT true, created_at TIMESTAMPTZ.
   - For `manualInputs`: Create a manual_inputs table with id UUID PK, input_id TEXT, period_month INT, period_year INT, value TEXT, updated_at TIMESTAMPTZ, updated_by TEXT.
   - Add indexes: primary keys, period_month+period_year composite on import tables.
   - Add RLS policies: anon can SELECT all tables (read-only for dashboards), authenticated can INSERT/UPDATE/DELETE.

4. **Merge screens** via internal `mergeScreens(blueprint.screens, technical.bindings)`:
   - For each blueprint screen, create a GenerationScreen that contains the original screen data plus the resolved bindings for each section (matching by screenId + sectionIndex).
   - Flatten section + binding into a single object per section for the renderer.

5. **Assemble and return GenerationManifest** with all fields populated.

---

Create `tools/wireframe-builder/lib/skill-renderer.ts` implementing renderSkillMd per research Pattern 3.

**renderSkillMd(manifest) -> string:**

The output SKILL.md must be completely self-contained (no external file references). Structure:

1. `# Skill -- {clientLabel} BI System` header with meta info (generated date, schema version).

2. `## Identidade` -- Client name, slug, what this system is.

3. `## Stack` -- Next.js 16, Supabase, Tailwind CSS, shadcn/ui, recharts (fixed stack per user decisions).

4. `## Supabase Schema` -- Full CREATE TABLE statements (from manifest.supabaseSchema.tables), CREATE INDEX statements, RLS policy definitions. Each as a SQL code block. Ready-to-run migrations.

5. `## Data Layer` -- Report types with column mappings table, field definitions table, formula definitions with expressions, manual input definitions, settings tables, classification rules, thresholds. Use Markdown tables for density.

6. `## Branding` -- Colors (primary, secondary, accent as hex values), fonts (heading, body), logo URL. CSS var pattern for injection.

7. `## Screens` -- For each screen: title, icon, period type, filters, compare switch, then each section as a subsection with its type, visual properties from blueprint, and data bindings from technical config. Use structured Markdown (headers + tables + code blocks).

8. `## Implementation Rules` -- Constraints: use recharts for charts, shadcn/ui for UI, Tailwind for styling, Brazilian number format (1.234,56) for display, monthly period model, Clerk auth integration.

Use template literal sections joined by `\n\n---\n\n`. Aim for density -- tables over prose. The renderer should handle all 15 section types in the screen rendering. Use exhaustive switch/if-else over sectionType with a comment for each type.

Both files import types via `@tools/wireframe-builder/types/` path aliases.
