Updated code — add enabled filtering:
```typescript
const navigationFromRegistry: NavItem[] = MODULE_REGISTRY
  .filter(m => m.status !== 'coming-soon')
  .filter(m => m.enabled !== false)  // Phase 38: hide disabled modules
  .flatMap(m => m.navChildren ?? [])
```

The `enabled !== false` check means:
- `enabled: true` -> show (explicitly enabled)
- `enabled: undefined` -> show (backward compat for manifests without the field)
- `enabled: false` -> hide (explicitly disabled)

IMPORTANT: If Phase 38 implemented enabled state via a React Context (e.g.,
`useModuleState()` hook returning enabled IDs from localStorage), the sidebar
filtering MUST use that context instead of reading the static `enabled` field
from the manifest. This is because the user can toggle modules at runtime
(REG-04) and the sidebar must react to those toggles.

In that case, the approach would be:
```typescript
// Inside the Sidebar component function (not at module scope)
const { isEnabled } = useModuleState()  // or whatever Phase 38's hook is called

const navigationFromRegistry: NavItem[] = MODULE_REGISTRY
  .filter(m => m.status !== 'coming-soon')
  .filter(m => isEnabled(m.id))  // reads from context/localStorage
  .flatMap(m => m.navChildren ?? [])
```

Move the `navigationFromRegistry` computation inside the component body if
it needs to use a hook. Use `useMemo` to avoid recomputation on every render:
```typescript
const navItems = useMemo(() =>
  MODULE_REGISTRY
    .filter(m => m.status !== 'coming-soon')
    .filter(m => isEnabled(m.id))
    .flatMap(m => m.navChildren ?? []),
  [isEnabled]
)
```
