**Algorithm:**

1. Fetch all documents via `getAllDocuments()` from docs-service (already ordered by parent_path, order)
2. Group documents by their `badge` field ("Processo" or "Padroes")
3. Within each badge group, build a tree based on `parent_path` depth:
   - Documents where slug has 1 segment (e.g., `processo/index`) are top-level section headers
   - Documents where slug has 2 segments (e.g., `processo/visao-geral`) are direct children
   - Documents where slug matches an `index` pattern (e.g., `processo/fases/index`) become parent nodes with children
   - Documents at deeper paths (e.g., `processo/fases/fase1`) become children of their parent index

Key mapping rules:
- A doc with slug ending in `/index` becomes a section header with `href` pointing to `/{slug}`
- Its children are docs whose `parent_path` matches the slug's directory
- Ordering within a parent uses the `order` field from the database
- The `label` is the doc's `title`
- The `href` is `/{slug}` (prepend slash to slug)

**Special cases to handle:**
- The "Padroes" section contains docs with parent_paths in both `padroes/`, `ferramentas/`, and `ferramentas/techs/`
- Tech Radar (`ferramentas/tech-radar`) is a parent with children from `ferramentas/techs/` parent_path
- The badge field determines which top-level section a doc belongs to, NOT the slug prefix

**Hook signature:**

```typescript
export function useDocsNav(): NavItem[] {
  const [navItems, setNavItems] = useState<NavItem[]>([])

  useEffect(() => {
    getAllDocuments().then((docs) => {
      const tree = buildNavTree(docs)
      setNavItems(tree)
    }).catch(() => {
      // On error, return empty — Sidebar falls back to hardcoded nav
    })
  }, [])

  return navItems
}
```
