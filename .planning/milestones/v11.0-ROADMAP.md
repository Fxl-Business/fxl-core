### Phase Details

### Phase 134: Schema Foundation
**Goal**: A infraestrutura de banco de dados para audit logging esta no ar — tabela imutavel, indexada e com triggers automaticos capturando mudancas criticas
**Depends on**: Nothing (foundation phase)
**Requirements**: AUDIT-01, AUDIT-02, AUDIT-03
**Success Criteria** (what must be TRUE):
  1. A tabela audit_logs existe no Supabase com todos os campos enterprise (id, org_id, actor_id, actor_email, actor_type, action, resource_type, resource_id, resource_label, ip_address, user_agent, metadata JSONB, created_at)
  2. INSERT via Supabase client funciona mas UPDATE e DELETE retornam erro de permissao negada (RLS imutavel)
  3. Qualquer INSERT ou UPDATE em tasks aciona automaticamente um registro em audit_logs com before/after JSONB
  4. Qualquer INSERT ou UPDATE em tenant_modules aciona automaticamente um registro em audit_logs com before/after JSONB
  5. Os indexes compostos (org_id+created_at DESC, actor_id, action, resource_type, GIN em metadata) estao presentes e verificados via \d audit_logs

**Plans**:

#### Plan 134.1: Create audit_logs table with immutable RLS
**What**: Migration 025 — create audit_logs table, enable RLS with append-only policies (INSERT allowed, UPDATE/DELETE denied), add composite indexes
**File**: `supabase/migrations/025_audit_logs.sql`
**Steps**:
1. Create `public.audit_logs` table with columns:
   - `id` uuid PK DEFAULT gen_random_uuid()
   - `org_id` text NOT NULL — tenant scope
   - `actor_id` text NOT NULL — Clerk user ID who performed the action
   - `actor_email` text — human-readable actor identifier
   - `actor_type` text NOT NULL DEFAULT 'user' CHECK (actor_type IN ('user', 'system', 'trigger'))
   - `action` text NOT NULL CHECK (action IN ('create', 'update', 'delete', 'archive', 'restore', 'sign_in', 'sign_out', 'impersonate'))
   - `resource_type` text NOT NULL — e.g. 'task', 'tenant_module', 'tenant', 'user'
   - `resource_id` text — ID of the affected resource
   - `resource_label` text — human-readable label (task title, module name, etc.)
   - `ip_address` text — request origin IP
   - `user_agent` text — request user-agent
   - `metadata` jsonb NOT NULL DEFAULT '{}' — flexible payload (before/after snapshots, extra context)
   - `created_at` timestamptz NOT NULL DEFAULT now()
2. Enable RLS on audit_logs
3. Create RLS policies:
   - `audit_logs_insert` — FOR INSERT TO anon, authenticated WITH CHECK (true) — triggers and app layer can insert
   - `audit_logs_select` — FOR SELECT TO anon, authenticated USING (super_admin bypass OR org_id matches JWT claim) — same pattern as migration 009
   - NO UPDATE policy — UPDATE denied by default when RLS is enabled
   - NO DELETE policy — DELETE denied by default when RLS is enabled
4. Create composite indexes:
   - `idx_audit_logs_org_created` ON (org_id, created_at DESC) — primary query pattern
   - `idx_audit_logs_actor` ON (actor_id) — filter by who
   - `idx_audit_logs_action` ON (action) — filter by what
   - `idx_audit_logs_resource_type` ON (resource_type) — filter by resource type
   - `idx_audit_logs_metadata` GIN ON (metadata) — JSONB containment queries
5. Deploy migration via `mcp__supabase__apply_migration`
6. Verify table exists and RLS works:
   - INSERT test (should succeed)
   - UPDATE test (should fail with permission denied)
   - DELETE test (should fail with permission denied)
**Success**: SC1 (table exists with all fields), SC2 (immutable RLS), SC5 (indexes present)
**Estimated size**: ~80 lines SQL

#### Plan 134.2: Create audit triggers on tasks and tenant_modules
**What**: Add trigger functions and triggers that auto-log changes to audit_logs when tasks or tenant_modules are modified
**File**: Same migration `025_audit_logs.sql` (append to Plan 134.1, single migration file)
**Steps**:
1. Create function `fn_audit_tasks()` with SECURITY DEFINER (bypasses RLS to write to audit_logs):
   - On INSERT: insert audit_logs row with action='create', resource_type='task', resource_id=NEW.id, resource_label=NEW.title, metadata=jsonb_build_object('after', row_to_json(NEW)), actor_id=COALESCE(NEW.created_by, 'system'), actor_type='trigger', org_id=NEW.org_id
   - On UPDATE: insert audit_logs row with action='update', resource_type='task', resource_id=NEW.id, resource_label=NEW.title, metadata=jsonb_build_object('before', row_to_json(OLD), 'after', row_to_json(NEW)), actor_id=COALESCE(NEW.created_by, OLD.created_by, 'system'), actor_type='trigger', org_id=NEW.org_id
   - RETURN NEW
2. Create function `fn_audit_tenant_modules()` with SECURITY DEFINER:
   - On INSERT: action='create', resource_type='tenant_module', resource_id=NEW.module_id, resource_label=NEW.module_id, metadata with after snapshot, actor_id='system', actor_type='trigger', org_id=NEW.org_id
   - On UPDATE: action='update', resource_type='tenant_module', resource_id=NEW.module_id, metadata with before/after, actor_id='system', actor_type='trigger', org_id=NEW.org_id
   - RETURN NEW
3. Create triggers:
   - `trg_audit_tasks` AFTER INSERT OR UPDATE ON tasks FOR EACH ROW EXECUTE FUNCTION fn_audit_tasks()
   - `trg_audit_tenant_modules` AFTER INSERT OR UPDATE ON tenant_modules FOR EACH ROW EXECUTE FUNCTION fn_audit_tenant_modules()
4. Deploy combined migration via `mcp__supabase__apply_migration`
5. Verify triggers work:
   - INSERT a test task → check audit_logs has a 'create' entry with correct metadata
   - UPDATE the test task → check audit_logs has an 'update' entry with before/after
   - INSERT a tenant_module row → check audit_logs
   - UPDATE a tenant_module row → check audit_logs
6. Clean up test data
**Success**: SC3 (tasks trigger works), SC4 (tenant_modules trigger works)
**Estimated size**: ~60 lines SQL (appended to Plan 134.1)
**Note**: Both plans produce a SINGLE migration file (025_audit_logs.sql). Plan 134.2 content is appended after Plan 134.1 content. The split is logical (for review), not physical (no separate migration).

### Phase 135: Capture Layer
**Goal**: O service layer tem uma funcao de captura de eventos que nunca interrompe a operacao principal e taga automaticamente sessoes de impersonation
**Depends on**: Phase 134 (tabela audit_logs deve existir)
**Requirements**: CAPT-01, CAPT-04
**Success Criteria** (what must be TRUE):
  1. Chamar logAuditEvent() com payload valido insere um registro em audit_logs sem requerir try/catch no call site
  2. Se logAuditEvent() falhar (ex: rede indisponivel), a operacao principal continua normalmente e o erro aparece no Sentry sem propagar excecao
  3. Durante uma sessao de impersonation ativa, todos os logs gerados via logAuditEvent() contem impersonator_id no campo metadata
  4. Fora de uma sessao de impersonation, o campo impersonator_id nao aparece no metadata (sem poluicao de dados)
**Plans**: TBD

### Phase 136: Edge Function Instrumentation
**Goal**: Todas as acoes administrativas criticas e eventos de auth sao capturados automaticamente no audit log sem intervencao manual do operador
**Depends on**: Phase 135 (logAuditEvent() deve existir)
**Requirements**: CAPT-02, CAPT-03
**Success Criteria** (what must be TRUE):
  1. Criar, atualizar ou arquivar um tenant via admin-tenants gera um registro em audit_logs com action e resource_type corretos
  2. Adicionar ou remover um membro de org via admin-tenants gera um registro em audit_logs
  3. admin-users tem a infraestrutura de audit (helper importado, getSupabaseAdmin pronto) para instrumentar futuras mutations (atualmente read-only, sem mutations para auditar)
  4. Um evento de sign-in bem-sucedido gera um registro em audit_logs com IP e user-agent do request

### Phase 137: Query API
**Goal**: O sistema tem um endpoint seguro e tipado para consultar logs com filtros e paginacao — pronto para ser consumido pela UI
**Depends on**: Phase 134 (dados devem existir para consultar)
**Requirements**: QUERY-01, QUERY-02
**Success Criteria** (what must be TRUE):
  1. Um request GET para audit-logs sem JWT super_admin retorna 401 (acesso negado)
  2. Um request GET para audit-logs com JWT super_admin retorna pagina de logs com limit/offset funcionando corretamente (max 100 por request)
  3. Os filtros org_id, actor_id, action e resource_type reduzem o resultado a apenas os registros correspondentes quando aplicados
  4. queryAuditLogs() em audit-service.ts aceita parametros tipados e retorna AuditLog[] sem usar any

### Phase 138: Admin UI + Retention
**Goal**: Super admins podem visualizar, filtrar, inspecionar e exportar o historico completo de operacoes criticas, e o sistema gerencia automaticamente a retencao de logs antigos
**Depends on**: Phase 137 (query API deve existir), Phase 135 (impersonation context para exibicao correta)
**Requirements**: UI-01, UI-02, UI-03, UI-04, OPS-01
**Success Criteria** (what must be TRUE):
  1. A pagina /admin/audit-logs esta acessivel no AdminSidebar e exibe uma tabela paginada com timestamp, actor, action, resource, org e IP
  2. Aplicar qualquer combinacao de filtros (date range, action type, actor, resource type, org) atualiza a tabela sem recarregar a pagina
  3. Clicar em qualquer linha da tabela abre um Sheet lateral com todos os campos do log e, para UPDATEs, exibe o diff formatado de before/after
  4. O botao de export baixa um arquivo .csv com os registros atualmente filtrados via native Blob API
  5. A configuracao de retencao em Platform Settings aceita um valor em dias e o pg_cron job deleta automaticamente logs mais antigos que o limite configurado
**Plans**: 3 plans
