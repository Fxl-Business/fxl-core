       - Export type SpecFile = { filename: string; content: string }
       - Export function generateProductSpec(manifest: GenerationManifest): SpecFile[]
       - The function calls 6 internal render functions, each returning a SpecFile:
         a. renderProductOverview(manifest) -> 'product-spec.md': System identity, navigation table (screens with icons/period types), auth roles table (admin/editor/viewer permissions), API endpoints table
         b. renderDatabaseSchema(manifest) -> 'database-schema.sql': Reuse the SQL generation logic from skill-renderer.ts renderSupabaseSchema() but output raw SQL (not Markdown-wrapped). Include CREATE TABLE, CREATE INDEX, ALTER TABLE ENABLE RLS, CREATE POLICY statements.
         c. renderDataLayer(manifest) -> 'data-layer.md': Reuse renderDataLayer() structure from skill-renderer.ts. Report types with column mappings, field definitions, formula definitions, manual inputs, settings tables, classifications, thresholds.
         d. renderScreens(manifest) -> 'screens.md': Reuse renderScreens() + renderSectionBinding() + renderBlueprintVisual() from skill-renderer.ts.
         e. renderBranding(manifest) -> 'branding.md': Reuse renderBranding() from skill-renderer.ts, plus add Google Fonts loading instructions.
         f. renderUploadRules(manifest) -> 'upload-rules.md': Extract report types, column mappings with format specifications, BR normalization rules for each column type, accepted file formats.

       Key differences from renderSkillMd():
       - NO renderStack() section (global skills handle this -- per user decision)
       - NO renderImplementationRules() section (global skills handle this -- per user decision)
       - SQL output is raw SQL, not Markdown code blocks
       - Each file is self-contained with its own header
       - Product spec describes WHAT to build, not HOW (per user decision)

       For this task, implement enough content in each renderer to pass the test assertions. Full rich rendering happens in Plan 02.
