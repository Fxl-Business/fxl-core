<task type="auto">
  <name>Task 1: Implement all 6 renderers with rich content adapted from skill-renderer.ts</name>
  <files>tools/wireframe-builder/lib/spec-generator.ts</files>
  <action>
    Evolve the spec-generator.ts skeleton from Plan 01 into full production renderers. For each of the 6 files:

    **1. renderProductOverview(manifest) -> product-spec.md:**
    - Header with "# Product Spec -- {clientLabel} BI Dashboard"
    - System Identity section: client slug, label, type "Dashboard BI"
    - Navigation sidebar table: | Screen | Icon | Period Type | -- one row per manifest.screens entry
    - Auth Roles table: 3 rows (admin, editor, viewer) with columns: Role, Screens, Upload, Settings, User Management. Admin gets all, Editor gets all screens + upload but no settings/user mgmt, Viewer gets read-only screens.
    - API Endpoints table: GET /api/data/:screenId, POST /api/upload/:reportType, GET /api/settings/:tableId, PUT /api/settings/:tableId/:rowId, GET /api/users (admin only), POST /api/users/invite (admin only)
    - Do NOT include stack information (global skills handle this per user decision)

    **2. renderDatabaseSchema(manifest) -> database-schema.sql:**
    - Adapt renderSupabaseSchema() from skill-renderer.ts but output PURE SQL (no Markdown wrapping)
    - Output CREATE TABLE statements for each table in manifest.supabaseSchema.tables
    - Output CREATE INDEX statements for each index in manifest.supabaseSchema.indexes
    - Output ALTER TABLE ENABLE ROW LEVEL SECURITY + CREATE POLICY for each RLS policy
    - Add a header comment: "-- Database Schema for {clientLabel}\n-- Generated from GenerationManifest\n-- Apply via Supabase SQL Editor or migration"

    **3. renderDataLayer(manifest) -> data-layer.md:**
    - Adapt renderDataLayer() from skill-renderer.ts almost directly
    - Report types with full column mapping tables (sourceColumn, targetField, dataType, format)
    - Field definitions table (id, label, source, column, aggregation, filter)
    - Formula definitions table (id, label, expression, format, references)
    - Manual inputs table (if any)
    - Settings tables (if any)
    - Classification rules (if any)
    - Threshold definitions table (if any)

    **4. renderScreens(manifest) -> screens.md:**
    - Adapt renderScreens() from skill-renderer.ts
    - Per screen: title, id, icon, periodType, hasCompareSwitch
    - Per section: visual properties via renderBlueprintVisual() and data binding via renderSectionBinding()
    - MUST handle all 15 section types exhaustively (same as skill-renderer.ts)
    - Copy renderSectionBinding() and renderBlueprintVisual() helper functions from skill-renderer.ts into spec-generator.ts (do NOT import from skill-renderer -- spec-generator should be self-contained)

    **5. renderBranding(manifest) -> branding.md:**
    - Adapt renderBranding() from skill-renderer.ts
    - Add section: "## Google Fonts" with instruction to load headingFont and bodyFont via Google Fonts API link
    - Add section: "## Tailwind Config" with instructions for extending theme with brand colors
    - CSS variables use --brand-* prefix (per Phase 04 decision)

    **6. renderUploadRules(manifest) -> upload-rules.md:**
    - New renderer (does not exist in skill-renderer.ts)
    - Header: "# Upload Rules -- {clientLabel}"
    - For each reportType in manifest.dataLayer.reportTypes:
      - Report type label and ID
      - Period model and files per period
      - Column mapping table: sourceColumn -> targetField with dataType and format
      - BR normalization rules: for each column with format '1.234,56', document the conversion. For each column with format 'dd/mm/yyyy', document the date conversion.
    - Section: "## Accepted Formats" -- CSV and XLSX
    - Section: "## Validation Rules" -- period_month 1-12, period_year 4 digits, no duplicate uploads for same period+reportType

    **Key principles applied throughout:**
    - Product spec describes WHAT, not HOW (per user decision: no re-teaching React/NestJS/Clerk)
    - Each file is self-contained with its own header
    - No cross-file references between spec files
    - Tables preferred for data density (matches skill-renderer.ts pattern)
  </action>
