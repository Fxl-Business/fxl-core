# Phase 02.2: Evolucao de Blocos Disponiveis - Context

**Gathered:** 2026-03-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Transform Blocos Disponiveis from a flat component catalog into a detailed spec/prompt system capable of generating any wireframe component. Each of the 22 blocks gets its own full behavior spec. The gallery gains cross-links to specs. Screen recipes evolve to include layout hints. The spec files live in docs/ferramentas/blocos/ and are rendered as navigable pages in the sidebar.

</domain>

<decisions>
## Implementation Decisions

### Spec depth per block
- Full behavior spec: props table with types/required/defaults, visual description, all conditional states as separate sections (compareMode, empty, wide, etc.)
- Each state section describes WHAT changes visually when that state is active
- Sizing rules included: document default size and variant behavior, but Blueprint can override
- Specs are the source of truth for what the component DOES — gallery is the visual proof it works
- Dynamic values (labels, colors) are per-wireframe, not part of the spec match

### Spec structure
- One .md file per block in `docs/ferramentas/blocos/` (22+ files)
- `docs/ferramentas/blocos/index.md` serves as landing page with summary table + links to each block
- Old `docs/referencias/blocos-disponiveis.md` is removed — content migrates to the new structure
- All spec pages rendered in the sidebar under Ferramentas > Wireframe Builder > Blocos Disponiveis
- Each spec page appears as a navigable page in the app

### Gallery-spec sync
- Manual convention enforced via SKILL.md rule: when creating/altering a component, update BOTH the spec .md AND the gallery entry
- Cross-links in both directions: gallery cards link to "Ver spec completa", spec pages link to "Ver na galeria"
- Gallery receives only cross-links in this phase — no visual improvements (deferred to 02.4)
- Gallery status concept (available/pending) remains as-is — all 22 are currently available

### Screen recipes
- Recipes stay in the landing page (index.md), not in individual spec files
- Evolve from flat lists to include layout/grid hints (e.g., "4x KpiCardFull in grid 4 col, then CalculoCard + WaterfallChart in grid 2 col 50/50")
- Format is extensible — current 8 types are base recipes, new types added as new clients/dashboards emerge
- Recipes serve as prompts for assembling a new screen's Blueprint

### Claude's Discretion
- Exact frontmatter fields for spec pages (title, badge, description format)
- Order of sections within each spec file
- How to structure the sidebar navigation for 22+ spec pages (collapsible group or flat list)
- Exact wording of cross-link labels
- How to handle the "Pendentes" table in the old doc (all are now implemented)

</decisions>

<specifics>
## Specific Ideas

- Specs are NOT used to auto-generate components — components are pre-built. Specs document capabilities so Claude (or a human) knows exactly what the component does and how to use it
- The spec should be detailed enough that Claude can recreate the component from scratch without seeing existing code
- Gallery and specs reflect the same capabilities but not the same dynamic values (labels, colors vary per wireframe)
- Receitas de tela serve as prompts for assembling new screen Blueprints — they answer "quais blocos uso para tela tipo X?"

</specifics>
