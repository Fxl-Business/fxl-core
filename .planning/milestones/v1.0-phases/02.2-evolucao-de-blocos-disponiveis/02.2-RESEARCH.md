## Summary

This phase transforms a flat component catalog (`docs/referencias/blocos-disponiveis.md`) into a structured specification system where each of the 22 wireframe builder components gets its own dedicated .md page with full behavior specs. The work is primarily documentation and content migration with targeted code changes to Sidebar.tsx, ComponentGallery.tsx, and SKILL.md.

The existing codebase already supports everything needed: the docs-parser auto-discovers .md files via `import.meta.glob('/docs/**/*.md')`, frontmatter with `badge` drives the page header and search index, and the Sidebar component uses a hardcoded navigation array that can be extended with a collapsible children group. No new libraries, no new components, no architectural changes are required.

**Primary recommendation:** Create 22 individual spec .md files in `docs/ferramentas/blocos/` following a consistent template derived from the existing component implementations, add an index.md landing page, update Sidebar navigation with a collapsible "Blocos Disponiveis" group, add cross-links in the gallery, and update SKILL.md with the sync convention rule.

**When to use:** Every single spec file.

### [Another State, e.g., wide]

[Default size, responsive behavior, container constraints]

### Anti-Patterns to Avoid
- **Duplicating implementation details in specs:** Specs describe WHAT the component does and its visual behavior. Do NOT copy the .tsx code line-by-line. The goal is for someone reading the spec to understand the component's capabilities without seeing the code.
- **Inconsistent frontmatter:** Every spec page MUST use `badge: Ferramentas`. The docs-parser and search-index depend on this.
- **Putting recipes in individual spec files:** Recipes belong in the landing page only. Individual specs document one component's behavior.
- **Adding dynamic values to specs:** Labels, colors, and values vary per wireframe. Specs document capabilities (e.g., "supports compareMode") not specific data.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Page rendering for new .md files | Custom page components | docs-parser + import.meta.glob | Auto-discovers .md files, already handles frontmatter, custom tags |
| Search indexing | Manual search index entries | badge field in frontmatter | Automatically indexed by existing infrastructure |
| Collapsible sidebar groups | Custom accordion component | Existing NavSection component | Already supports depth-based collapsing with chevron toggle |
| Doc page routing | Manual route registration in App.tsx | Wildcard doc route | App.tsx already has catch-all route for docs |

**Key insight:** The entire docs infrastructure is already built. Creating new .md files in `docs/` is the only requirement for them to become navigable pages. The only manual step is adding sidebar navigation entries.

## Common Pitfalls

### Pitfall 1: File Naming vs Sidebar Href Mismatch
**What goes wrong:** Sidebar href `/ferramentas/blocos/kpi-card` does not match file path `docs/ferramentas/blocos/kpi-card.md`.
**Why it happens:** The docs-parser strips the `docs/` prefix and `.md` extension to generate paths. The file at `docs/ferramentas/blocos/kpi-card.md` becomes accessible at `/ferramentas/blocos/kpi-card`.
**How to avoid:** Always verify: `href` in Sidebar = file path minus `docs/` prefix minus `.md` extension.
**Warning signs:** 404 on sidebar click, blank page.

### Pitfall 2: Incomplete Prop Documentation
**What goes wrong:** A spec omits a prop that exists in the component, making the spec unreliable as a prompt.
**Why it happens:** Components have implicit props (e.g., `compareMode` on KpiCardFull changes both variation and semaforo visibility).
**How to avoid:** Read the actual .tsx implementation for each component before writing its spec. Every prop must be documented.
**Warning signs:** Gallery shows props not mentioned in the spec.

### Pitfall 3: Sidebar Becoming Too Long
**What goes wrong:** 22 spec page links make the sidebar unwieldy when expanded.
**Why it happens:** Flat listing all components without grouping.
**How to avoid:** Use the collapsible parent pattern. Consider grouping by category (Cards, Charts, Tables, Layout, Inputs) as the gallery already does.
**Warning signs:** Users scroll excessively in sidebar.

### Pitfall 4: Breaking the Old Reference Link
**What goes wrong:** After removing `docs/referencias/blocos-disponiveis.md`, existing links to `/referencias/blocos-disponiveis` break (including in Sidebar, wireframe-builder.md, ferramentas/index.md, and SKILL.md).
**Why it happens:** Incomplete link migration.
**How to avoid:** Search the entire codebase for `/referencias/blocos-disponiveis` and update all references.
**Warning signs:** 404 errors on previously working links.

### Pitfall 5: Forgetting to Update wireframe-builder.md Landing Page
**What goes wrong:** The wireframe-builder landing page still uses a phase-card pointing to the old blocos-disponiveis path.
**Why it happens:** Only updating the sidebar but not the .md content.
**How to avoid:** Update the phase-card href in `docs/ferramentas/wireframe-builder.md` to point to the new index page.

### Example 1: Spec File for KpiCardFull (representative)

```markdown
---
title: KpiCardFull
badge: Ferramentas
description: Card de KPI expandido com sparkline, semaforo e modo comparacao
---

# KpiCardFull

Card de indicador financeiro com valor principal, subtitulo, sparkline SVG embutido
e semaforo opcional (verde/amarelo/vermelho). Suporta modo comparacao que revela
variacao temporal e semaforo simultaneamente.

[Ver na galeria](/ferramentas/wireframe-builder/galeria)

## Props

| Prop | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| label | string | Yes | - | Titulo do KPI em uppercase tracking-wider |
| value | string | Yes | - | Valor principal formatado (ex: "R$ 125.000") |
| sub | string | No | - | Subtitulo abaixo do valor (ex: "% s/ faturamento") |
| variation | string | No | - | Texto de variacao (ex: "+12,3% vs Mar/26") |
| variationPositive | boolean | No | true | Cor da variacao: verde se true, vermelho se false |
| semaforo | 'verde' \| 'amarelo' \| 'vermelho' | No | - | Status visual com dot colorido |
| semaforoLabel | string | No | Nome da cor | Label ao lado do dot (ex: "Dentro da meta") |
| sparkline | number[] | No | - | Array de valores para mini grafico SVG |
| wide | boolean | No | false | Se true, ocupa 2 colunas no grid (col-span-2) |
| compareMode | boolean | No | false | Ativa exibicao de variacao e semaforo |

## Visual Description

Card branco arredondado com borda cinza e sombra sutil. Layout vertical:
1. Label em uppercase 10px tracking-wider cinza-400
2. Valor em 2xl bold cinza-800
3. Sub em xs cinza-400 (quando presente)
4. Semaforo: dot 2x2 colorido + label em xs (quando compareMode=true E semaforo presente)
5. Variacao: badge inline com fundo verde-50/vermelho-50 (quando compareMode=true E variation presente)
6. Sparkline: SVG polyline h-8 largura total em cinza-400 (quando sparkline presente)

## Conditional States

### compareMode
Quando `compareMode=true`:
- Semaforo aparece (dot colorido + label)
- Variacao aparece (badge com triangulo)
Quando `compareMode=false`:
- Semaforo oculto
- Variacao oculta
Sparkline sempre visivel independente do compareMode.

### wide
Quando `wide=true`: card recebe `col-span-2` e ocupa largura dobrada no grid.
Quando `wide=false`: card ocupa 1 coluna.

## Sizing Rules

- Largura: controlada pelo grid pai (tipicamente `grid-cols-4`)
- Altura: auto, determinada pelo conteudo
- Sparkline: h-8, largura 100% do card
- wide=true: col-span-2
