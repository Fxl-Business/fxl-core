**Researched:** 2026-03-08

## Summary

This phase is a purely visual overhaul of the FXL Core application. The codebase already has the correct infrastructure: Tailwind CSS 3 with `darkMode: ['class']` configured, shadcn/ui with CSS variables enabled (`cssVariables: true` in components.json), and HSL-based CSS custom properties in globals.css. The work is about replacing the current fxl-navy (#1B3A5C) color system with a new palette inspired by Supabase docs, introducing gold as the accent color, adding a complete dark mode theme, and bringing all pages up to production-grade visual quality.

The codebase audit reveals 35 occurrences of `fxl-navy/fxl-blue` colors across 15 files, plus 27 occurrences of hardcoded slate-* classes across 7 files. The work is systematic: define new CSS variables (light + dark), update the Tailwind config to remove the `fxl.*` namespace, then sweep through every component replacing hardcoded colors with semantic tokens. The ComponentGallery and wireframe viewer pages have extensive hardcoded gray-* usage that also needs migration.

**Primary recommendation:** Start with the design token foundation (CSS variables in globals.css + tailwind.config.ts), then propagate outward: layout shell (TopNav, Sidebar, Layout), doc components (prose styles, PageHeader, Callout, etc.), pages (Home, DocRenderer, client pages), and finally the ComponentGallery. Dark mode toggle should be added to TopNav alongside the search command.

## Architecture Patterns

### Design Token Architecture (CSS Variables)

The core pattern is a two-layer token system:

**Layer 1: CSS Variables (globals.css)**
```css
/* Light mode (default) */
:root {
  --background: 210 20% 98%;
  --foreground: 220 20% 10%;
  --primary: 220 16% 22%;          /* cool dark gray-blue, inspired by Supabase */
  --primary-foreground: 0 0% 100%;
  --accent: 43 96% 56%;            /* gold accent */
  --accent-foreground: 43 50% 12%;
  --muted: 220 14% 96%;
  --muted-foreground: 220 10% 46%;
  --sidebar: 220 16% 98%;
  --sidebar-foreground: 220 16% 22%;
  --sidebar-accent: 43 96% 56%;
  /* ... all tokens */
}

/* Dark mode */
.dark {
  --background: 220 20% 7%;
  --foreground: 210 20% 96%;
  --primary: 210 20% 96%;
  --primary-foreground: 220 20% 7%;
  --accent: 43 90% 50%;
  --accent-foreground: 43 20% 96%;
  --sidebar: 220 16% 10%;
  /* ... inverted tokens */
}
```

**Layer 2: Tailwind Config (semantic mapping)**
```typescript
// tailwind.config.ts -- remove fxl.* namespace, keep semantic tokens
colors: {
  border: 'hsl(var(--border))',
  background: 'hsl(var(--background))',
  foreground: 'hsl(var(--foreground))',
  primary: { DEFAULT: 'hsl(var(--primary))', foreground: 'hsl(var(--primary-foreground))' },
  accent: { DEFAULT: 'hsl(var(--accent))', foreground: 'hsl(var(--accent-foreground))' },
  // NO more fxl.navy, fxl.blue etc.
}
```

### Component Migration Pattern

Every component follows the same migration:
```
BEFORE: className="text-fxl-navy" or className="text-slate-900"
AFTER:  className="text-foreground" or className="text-primary"

BEFORE: className="bg-fxl-navy"
AFTER:  className="bg-primary"

BEFORE: className="text-slate-500"
AFTER:  className="text-muted-foreground"

BEFORE: className="border-slate-200"
AFTER:  className="border-border"
```

### Dark Mode Toggle Pattern

```typescript
// Simple localStorage-persisted toggle using class strategy
function ThemeToggle() {
  const [dark, setDark] = useState(() => {
    if (typeof window === 'undefined') return false
    return document.documentElement.classList.contains('dark')
  })

  function toggle() {
    const next = !dark
    setDark(next)
    document.documentElement.classList.toggle('dark', next)
    localStorage.setItem('theme', next ? 'dark' : 'light')
  }

  // On mount: check localStorage, then system preference
  useEffect(() => {
    const saved = localStorage.getItem('theme')
    if (saved === 'dark') {
      document.documentElement.classList.add('dark')
      setDark(true)
    }
  }, [])

  return (
    <button onClick={toggle}>
      {dark ? <Sun /> : <Moon />}
    </button>
  )
}
```

### Prose Styling Pattern (Markdown Content)

The prose classes in globals.css must use semantic tokens, not hardcoded colors:

```css
/* BEFORE */

.prose th { @apply bg-fxl-navy text-white; }

/* AFTER */
.prose h1 { @apply text-foreground; }
.prose th { @apply bg-muted text-foreground; }
```

### Anti-Patterns to Avoid
- **Hardcoded colors in components:** Never use `text-slate-900`, `bg-gray-50`, etc. Always use semantic tokens (`text-foreground`, `bg-muted`). Exception: status colors (green/amber/red for badges) which are semantic by nature.
- **Inline style colors:** MarkdownRenderer.tsx has inline styles with hardcoded hex values (`#0f172a`, `#1e293b`, `#e2e8f0`). These must become theme-aware CSS classes.
- **Opacity on semantic tokens:** `bg-fxl-navy/10` works but `bg-primary/10` needs HSL format without the `hsl()` wrapper in the CSS variable (which is already the case -- the current format `215 72% 24%` supports opacity modifiers).
- **Mixing fxl.* and semantic tokens:** During migration, never leave both systems in place in the same component. Complete the swap per-component.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Dark mode toggle state | Custom context + provider | localStorage + classList.toggle('dark') + Tailwind class strategy | The Tailwind `darkMode: ['class']` already handles everything. No React context needed -- just toggle the class on `<html>`. |
| Color palette generation | Manual HSL calculations | shadcn/ui color system conventions | The background/foreground pattern with HSL channels is battle-tested. Follow the convention. |
| Theme persistence | Custom state management | localStorage('theme') read on mount | Simple, no dependencies, SSR-safe with the mount check pattern. |
| Component variant styling | Custom if/else chains | class-variance-authority (CVA) | Already installed and used in button.tsx and badge.tsx. Use it for any component with visual variants. |
| Prose/markdown dark mode | Separate dark markdown styles | Tailwind prose semantic tokens | Using `text-foreground`, `bg-muted` etc. automatically adapts to dark mode. No separate dark prose stylesheet needed. |

**Key insight:** This is a token replacement exercise, not a component rewrite. The architecture (shadcn/ui + Tailwind + CSS variables) already supports everything needed. The work is defining the right tokens and systematically replacing hardcoded values.

## Common Pitfalls

### Pitfall 1: Forgetting the MarkdownRenderer inline styles
**What goes wrong:** Code blocks and inline code in rendered markdown keep light-mode colors in dark mode because MarkdownRenderer.tsx uses inline `style={{}}` with hardcoded hex values.
**Why it happens:** The component uses React inline styles instead of Tailwind classes to style `<pre>`, `<code>` elements.
**How to avoid:** Replace all inline styles in MarkdownRenderer.tsx with Tailwind classes that use semantic tokens. The `pre` block uses `background: '#0f172a'` which should become a CSS class like `bg-slate-900 dark:bg-slate-950` or better yet a custom `--code-bg` variable.
**Warning signs:** Code blocks look fine in light mode but jarring in dark mode.

### Pitfall 2: Clerk components not following theme
**What goes wrong:** Clerk's SignIn, SignUp, and UserProfile components don't pick up the new theme because they use their own styling system.
**Why it happens:** Clerk uses `@clerk/ui/themes/shadcn` which reads CSS variables, but if variable names change or new custom variables are added, Clerk components may not reflect them.
**How to avoid:** The import `@clerk/ui/themes/shadcn.css` in globals.css and `appearance: { theme: shadcn }` in ClerkProvider should continue working because they read standard CSS variables (--background, --primary, etc.). Verify after theme change that Clerk components look correct.
**Warning signs:** Login page looks different from the rest of the app.

### Pitfall 3: The ComponentGallery has 60+ hardcoded gray references
**What goes wrong:** The ComponentGallery page (650 lines) uses gray-100, gray-200, gray-300, gray-400, gray-500, gray-600, gray-800 extensively for toolbar backgrounds, borders, text, and status badges.
**Why it happens:** It was built independently from the main theme system, using direct Tailwind color classes.
**How to avoid:** Systematically replace: gray-800 -> foreground/primary, gray-500 -> muted-foreground, gray-200 -> border, gray-100 -> muted, gray-50 -> background. But status badges (green-50/green-700, amber-50/amber-700) should stay -- they are semantic status colors.
**Warning signs:** Gallery looks washed out or has zero contrast in dark mode.

### Pitfall 4: Background gradient in Layout.tsx
**What goes wrong:** The Layout component uses a hardcoded CSS gradient `bg-[linear-gradient(180deg,#f7fafc_0%,#ffffff_28%,#f4f7fb_100%)]` which will look wrong in dark mode.
**Why it happens:** Complex gradients can't use CSS variables directly in Tailwind's arbitrary value syntax.
**How to avoid:** Either: (a) replace with a simple `bg-background` for clean look, or (b) define the gradient as a CSS class in globals.css with both light and dark variants.
**Warning signs:** White gradient flash when dark mode is active.

### Pitfall 5: Inconsistent color usage across src/pages vs src/components
**What goes wrong:** Some files use `text-slate-900` (Home.tsx, DocPageHeader.tsx), others use `text-foreground` (DocBreadcrumb.tsx), others use `text-fxl-navy` (PageHeader.tsx). After migration, stray hardcoded colors remain.
**Why it happens:** The codebase grew organically with different conventions at different times.
**How to avoid:** After the migration, run `grep -r "text-slate-\|bg-slate-\|text-gray-\|bg-gray-\|text-fxl-\|bg-fxl-" src/` and verify zero results (except for semantic status colors like green, amber, red).
**Warning signs:** Inconsistent text colors between pages.

### Pitfall 6: WireframeViewer and SharedWireframeView use inline styles
**What goes wrong:** These full-screen pages (WireframeViewer.tsx ~140 lines, SharedWireframeView.tsx ~270 lines) use React inline `style={{}}` objects extensively with hardcoded colors (#F5F5F5, #212121, #757575, #424242, etc.).
**Why it happens:** They were built as isolated full-screen experiences outside the Layout component.
**How to avoid:** These pages are in scope for VISUAL-02 but are lower priority. They should be migrated to Tailwind classes. However, since they render wireframes that have their own visual identity (client-facing), the wireframe preview area itself may intentionally keep its own styling.
**Warning signs:** Full-screen wireframe viewer breaks visually in dark mode.

### Dark Mode Toggle Component

```typescript
// Source: Tailwind docs dark mode class strategy
import { Moon, Sun } from 'lucide-react'
import { useEffect, useState } from 'react'
import { Button } from '@/components/ui/button'

export function ThemeToggle() {
  const [dark, setDark] = useState(false)

  useEffect(() => {
    const saved = localStorage.getItem('theme')
    const prefersDark = window.matchMedia('(prefers-color-scheme: dark)').matches
    const isDark = saved === 'dark' || (!saved && prefersDark)
    setDark(isDark)
    document.documentElement.classList.toggle('dark', isDark)
  }, [])

  function toggle() {
    const next = !dark
    setDark(next)
    document.documentElement.classList.toggle('dark', next)
    localStorage.setItem('theme', next ? 'dark' : 'light')
  }

  return (
    <Button variant="ghost" size="icon" onClick={toggle} className="h-8 w-8">
      {dark ? <Sun className="h-4 w-4" /> : <Moon className="h-4 w-4" />}
    </Button>
  )
}
```
