2. **## Setup Sentry** ŌĆö Complete setup for a spoke frontend:
   - Step 1: Install `@sentry/react`
   - Step 2: Create Sentry project, get DSN
   - Step 3: Add env var `VITE_SENTRY_DSN` to `.env` and `.env.example`
   - Step 4: Add optional `VITE_SENTRY_ENVIRONMENT` (defaults to 'production')
   - Show the exact initialization code from Nexo's `src/main.tsx`:
     ```typescript
     const sentryDsn = import.meta.env.VITE_SENTRY_DSN
     if (sentryDsn) {
       Sentry.init({
         dsn: sentryDsn,
         environment: import.meta.env.VITE_SENTRY_ENVIRONMENT ?? 'production',
         integrations: [
           Sentry.browserTracingIntegration(),
           Sentry.replayIntegration({ maskAllText: false, blockAllMedia: false }),
         ],
         tracesSampleRate: 0.2,
         replaysSessionSampleRate: 0,
         replaysOnErrorSampleRate: 1.0,
       })
     }
     ```
   - Explain each config: tracesSampleRate 0.2 = 20% of requests traced, replaysOnErrorSampleRate 1.0 = 100% replay on errors, replaysSessionSampleRate 0 = no random session replays
   - Add callout warning: `{% callout type="warning" %}O Sentry so inicializa se VITE_SENTRY_DSN estiver definido. Em desenvolvimento local sem DSN, nenhum erro e enviado ŌĆö isso e intencional.{% /callout %}`

3. **## Contexto de Usuario e Org** ŌĆö Setting user and org context:
   - Show the `SentryContextSetter` pattern from `src/App.tsx`:
     ```typescript
     function SentryContextSetter() {
       const { userId } = useAuth()
       const { activeOrg } = useActiveOrg()
       useEffect(() => {
         if (userId) { Sentry.setUser({ id: userId }) }
         else { Sentry.setUser(null) }
       }, [userId])
       useEffect(() => {
         if (activeOrg?.id) { Sentry.setTag('org_id', activeOrg.id) }
         else { Sentry.setTag('org_id', undefined) }
       }, [activeOrg?.id])
       return null
     }
     ```
   - Explain: renders nothing, purely side-effect component, placed inside auth providers in App.tsx
   - For a spoke: adapt to use the spoke's auth hook (still Clerk) and org context
   - Note: `Sentry.setTag('org_id', ...)` allows filtering errors by tenant in Sentry dashboard

4. **## Captura de Erros por Modulo** ŌĆö Module-scoped error capture:
   - Reference the error boundary pattern (link to `/sdk/error-boundaries`)
   - Show the `componentDidCatch` from ModuleErrorBoundary:
     ```typescript
     Sentry.captureException(error, {
       tags: { module: this.props.moduleName },
       contexts: { react: { componentStack: errorInfo.componentStack ?? '' } },
     })
     ```
   - Explain: module tag allows filtering errors by module in Sentry, componentStack shows React tree

5. **## Captura Manual** ŌĆö Manual error capture for async/event handler errors:
   - Pattern: `try { ... } catch (err) { Sentry.captureException(err, { tags: { context: 'token-exchange' } }) }`
   - When to use: fetch errors, async operations, event handlers (error boundaries don't catch these)
   - Always include a descriptive tag for filtering

6. **## Source Maps** ŌĆö Source map upload for readable stack traces:
   - Explain that without source maps, Sentry shows minified code
   - Add Vite plugin: `@sentry/vite-plugin`
   - Configuration in `vite.config.ts`:
     ```typescript
     import { sentryVitePlugin } from '@sentry/vite-plugin'
     // In plugins array:
     sentryVitePlugin({
       org: 'your-org',
       project: 'your-project',
       authToken: process.env.SENTRY_AUTH_TOKEN,
     })
     ```
   - Add `SENTRY_AUTH_TOKEN` to CI env vars
   - Add `build.sourcemap: true` to vite config

7. **## Alertas Basicos** ŌĆö Basic alert setup in Sentry dashboard:

   - Create alert for "Error count > 10 in 1 hour" for high-frequency issues
   - Recommend: start with email alerts, add Slack/Discord later
   - Tag-based alerts: `module:Tarefas` to route module-specific alerts
