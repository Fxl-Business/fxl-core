1. **Title + intro callout** — `# Retry com Backoff`
   - Add callout: `{% callout type="info" %}Este padrao foi introduzido no Nexo v9.0 (Resiliencia de Plataforma).{% /callout %}`
   - Explain the problem: network calls fail transiently (DNS hiccups, server overload, cold starts). Without retry, a single failure breaks the user flow. With retry + exponential backoff, transient failures are absorbed transparently.

3. **## Configuracao** — Explain each option:
   - `maxRetries` (default 3) — Number of retry attempts after initial failure. Total calls = maxRetries + 1.
   - `baseDelay` (default 1000ms) — Delay before first retry.
   - `backoffFactor` (default 2) — Multiplier: delay = baseDelay * backoffFactor^attempt. Show the progression: 1s -> 2s -> 4s.
   - `signal` (optional AbortSignal) — Cancels retry chain immediately. Used for org switch scenarios.
   - `isRetryable` (optional predicate) — Custom function to decide if error should be retried. Default: `isNetworkOrServerError` (retries TypeError + 5xx, never 4xx).
   - Add a table showing the delay progression:

     | Attempt | Delay (default config) |
     |---------|----------------------|
     | 1 (initial) | 0ms (immediate) |
     | 2 (retry 1) | 1000ms |
     | 3 (retry 2) | 2000ms |
     | 4 (retry 3) | 4000ms |

4. **## Quando Usar** — Guidelines for where to apply:
   - Token exchange (critical — user can't use the app without it)
   - API calls to edge functions (network can fail)
   - Any fetch to external services
   - NOT for: user input validation, local state operations, file reads

5. **## Exemplos de Uso** — Real call site examples:

   **Exemplo 1: Token Exchange (com AbortSignal)**
   ```typescript
   const result = await withRetry(
     () => exchangeToken(clerkToken, activeOrg.id, signal),
     { maxRetries: 3, baseDelay: 1000, backoffFactor: 2 },
   )
   ```
   Explain: token exchange is the most critical call — without it, RLS can't scope data. AbortSignal cancels retries when user switches org.

   **Exemplo 2: Service Layer (admin-service.ts)**
   ```typescript
   export async function listUsers(): Promise<AdminUserListResponse> {
     return withRetry(async () => {
       const res = await fetch(`${FUNCTIONS_URL}/admin-users`, {
         headers: await getAuthHeaders(),
       })
       if (!res.ok) throw new Error(`Fetch failed (${res.status})`)
       return res.json()
     })
   }
   ```
   Explain: wrapping the entire fetch-parse cycle. The error message includes `(status)` so `isNetworkOrServerError` can extract the status code and decide if it's retryable (5xx yes, 4xx no).

   **Exemplo 3: Custom isRetryable**
   ```typescript
   await withRetry(fetchExternalData, {
     isRetryable: (error) => {
       // Retry rate limits (429) in addition to defaults
       if (error.message.includes('(429)')) return true
       return isNetworkOrServerError(error)
     },
   })
   ```

6. **## Integracao com AbortController** — Explain the abort pattern:
   - When to use: org switch, component unmount, user cancellation
   - Show the pattern: create AbortController, pass signal to withRetry, abort on cleanup
   - Reference OrgTokenContext as the canonical example

7. **## Anti-Patterns** — What NOT to do:
   - Don't retry user authentication errors (401/403) — they won't succeed on retry
   - Don't retry validation errors (400) — fix the input instead
   - Don't set maxRetries too high (> 5) — user waits too long
   - Don't use without the error status pattern `throw new Error(\`...(${status})\`)` — isNetworkOrServerError can't parse the status

8. **## Testes** — Testing guidance:
   - Show how to test with vitest fake timers (reference from retry.test.ts)
   - Key test cases: success on first try, retry on network error, no retry on 4xx, abort during retry, custom isRetryable
