Estrutura do conteudo (todas as secoes obrigatorias):

**1. Introducao** — Paragrafo explicando que todo spoke FXL deve manter documentacao padronizada. A documentacao serve dois consumidores: humanos (desenvolvedores) e IA (Claude Code). Callout: "Documentacao nao e opcional — e requisito de qualidade."

**2. README.md Obrigatorio** — Estrutura minima do README para todo spoke:
- Secoes obrigatorias em tabela:
  - `# Nome do Projeto` — nome legivel
  - `## Sobre` — natureza do projeto (1-2 frases), App ID, FXL Contract version
  - `## Stack` — tabela de tecnologias com versoes
  - `## Setup` — passo a passo para rodar localmente (clone, install, env, dev)
  - `## Variaveis de Ambiente` — tabela com cada var, descricao, e onde obter
  - `## Scripts` — tabela com cada script npm e descricao
  - `## Deploy` — como deployar (Vercel, branch strategy)
  - `## Estrutura` — arvore de diretorios resumida
- Exemplo completo de README minimo em code block
- Callout: "O README deve permitir que um novo desenvolvedor rode o projeto em < 10 minutos"

**3. CLAUDE.md por Projeto** — Guia de como estruturar o CLAUDE.md para cada spoke:
- Proposito: e o arquivo mais importante para operacao com IA — define o que o Claude Code sabe sobre o projeto
- Secoes obrigatorias (tabela):
  - Natureza do projeto (o que e, App ID, contract version)
  - Stack (tecnologias usadas)
  - Estrutura do projeto (arvore atualizada)
  - Entidades do dominio (lista com campos)
  - Convencoes de codigo (naming, imports, commits)
  - Regras de seguranca (env vars, RLS, auth)
  - Contract endpoints (tabela dos 6 endpoints)
  - Quality gates (comandos de validacao)
  - O que nunca fazer (anti-patterns)
- Referencia ao `CLAUDE.md.template` em templates.md para geracao automatica
- Callout: "Manter o CLAUDE.md atualizado apos cada mudanca estrutural — e a fonte de verdade para o Claude Code"

**4. Changelog** — Como manter historico de mudancas:
- Formato: seguir a convencao de commit do projeto (tipo(escopo): mensagem)
- Tipos de commit:
  - `docs:` — alteracoes em documentacao
  - `app:` — alteracoes no codigo da aplicacao
  - `infra:` — alteracoes de infraestrutura/config
  - `tool(nome):` — alteracoes em tools
  - `[client-slug]:` — alteracoes especificas de cliente
- Quando atualizar: a cada PR mergeado
- Formato do CHANGELOG.md (se usado):
  ```
  ## [data] - descricao da release
  ### Added
  ### Changed
  ### Fixed
  ```
- Callout: "Para projetos que usam GSD, o historico de milestones/phases em .planning/ substitui o CHANGELOG.md tradicional"

**5. Documentacao de API** — Como documentar endpoints do FXL Contract no spoke:
- Cada endpoint deve ter: metodo + rota, descricao, request params/body, response schema, exemplo de request/response
- Formato recomendado com code block mostrando exemplo:
  ```
  ### GET /api/fxl/entities/:type

  Retorna lista de entidades de um tipo especifico.

  **Params:** `:type` — tipo da entidade (ex: `reservations`)
  **Query:** `?limit=N` — limite de resultados (default: 50)

  **Response:**
  {json com exemplo}
  ```
- Referencia ao contract.md para detalhes do contract padrao
- Callout: "Documentar apenas os endpoints do FXL Contract e endpoints customizados do spoke — nao re-documentar o Supabase client"

**6. Diagramas de Arquitetura** — Quando e como criar:
- Usar Mermaid syntax para diagramas inline em markdown
- Quando criar: ao adicionar integracao nova, ao mudar fluxo de dados, ao criar modulo novo
- Exemplo de diagrama Mermaid simples mostrando fluxo spoke -> hub
- Callout: "Diagramas sao opcionais — priorizar documentacao textual clara"

**7. Checklist de Documentacao** — Checklist final para validar que o spoke tem documentacao completa:
- [ ] README.md com todas as secoes obrigatorias
- [ ] CLAUDE.md atualizado com estrutura e entidades
- [ ] Endpoints do FXL Contract documentados
- [ ] Variaveis de ambiente documentadas em .env.example
- [ ] Scripts npm documentados no README
- [ ] Convencoes de commit seguidas
