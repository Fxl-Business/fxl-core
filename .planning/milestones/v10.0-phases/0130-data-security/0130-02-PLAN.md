3. **## JWT Bridge: Clerk -> Supabase** — Token exchange pipeline:
   - **Problema:** Clerk usa RS256 (JWKS), Supabase usa HS256 (JWT secret). Sao incompativeis.
   - **Solucao:** Edge Function que recebe Clerk token, valida via JWKS, gera JWT HS256 com claims org_id e super_admin para o Supabase.
   - Show the token exchange flow diagram (text-based):
     ```
     Frontend (Clerk) -> Edge Function (auth-token-exchange) -> Supabase (JWT com org_id)
     session.getToken() -> POST /auth-token-exchange -> access_token (HS256)
     ```
   - Show real code from `token-exchange.ts`: the `exchangeToken()` function signature, headers, body with org_id
   - Show real code from `OrgTokenContext.tsx`: how the provider manages token lifecycle:
     - Exchange on mount and on org switch
     - AbortController cancels in-flight requests when org changes
     - withRetry wrapping for resilience (maxRetries: 3, baseDelay: 1000)
     - Token override for impersonation
   - Edge function deployment: MUST use `verify_jwt: false` because gateway JWT check blocks Clerk tokens (PITFALL #9)
   - Edge function secrets: use `JWT_SIGNING_SECRET` (not `SUPABASE_JWT_SECRET` — PITFALL #11)

4. **## RLS por Tenant** — Row Level Security by org_id:
   - Brief overview of the three-tier pattern (reference database.md for full details)
   - How JWT claims reach Supabase: `current_setting('request.jwt.claims', true)` -> parse as jsonb -> extract `org_id`
   - Cross-reference: "Para o padrao completo de RLS em tres niveis, veja [Banco de Dados](/sdk/database#row-level-security-rls)."
   - Scope-based access for documents: product docs (shared), tenant docs (isolated by org_id)
   - Reminder: `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` is mandatory — without it, RLS policies are ignored

5. **## Protecao de Environment Variables** — Env var security:
   - Rule: `VITE_` prefix exposes variable to the browser (Vite injects into bundle)
   - Secrets (service role key, API keys, JWT signing secret) NEVER get `VITE_` prefix
   - `.env.local` never committed — `.env.example` committed with placeholder values
   - Required env vars for every spoke:
     ```
     VITE_SUPABASE_URL=https://xxx.supabase.co
     VITE_SUPABASE_ANON_KEY=eyJ...
     VITE_CLERK_PUBLISHABLE_KEY=pk_live_...
     FXL_API_KEY=<chave-gerada>            # server-side only, SEM prefixo VITE_
     ```
   - Edge function secrets: use Supabase Dashboard or `supabase secrets set` (not SUPABASE_ prefix — PITFALL #11)

6. **## ConnectorResult\<T\> — Validacao de APIs Externas** — The FXL standard for external API calls:
   - Show the full `ConnectorResult<T>` discriminated union:
     ```typescript
     type ConnectorResult<T> =
       | { ok: true; data: T }
       | { ok: false; error: ConnectorError }
     ```
   - Show the `ConnectorError` type with all 5 variants: offline, unauthorized, server-error, invalid-manifest, unknown
   - Show the pattern for using it:
     ```typescript
     const result = await fetchManifest(baseUrl)
     if (!result.ok) {
       // Handle error by type
       switch (result.error.type) {
         case 'offline': // show offline state
         case 'unauthorized': // retry with refresh, then show reconnect
         case 'server-error': // show error in card, don't propagate
       }
       return
     }
     // result.data is typed as T
     ```
   - This pattern is the FXL standard for ALL external API calls, not just the connector module
   - Explain WHY: never trust external API response shapes; always validate before accessing fields

7. **## Regras de Seguranca em APIs** — API safety rules from PITFALLS.md:
   - **Promise.allSettled para fetches independentes (PITFALL #1):**
     Show the wrong way (Promise.all) and right way (Promise.allSettled) with real example from admin dashboard
   - **Null safety com fallbacks (PITFALL #6):**

   - **Formato de resposta nao e consistente (PITFALL #3):**
     Never assume endpoints follow the same response format — verify each one

   - **Timeout em chamadas externas:**
