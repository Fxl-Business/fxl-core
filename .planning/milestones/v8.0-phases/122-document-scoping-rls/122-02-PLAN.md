Para testar RLS com JWT customizado, criar um Supabase client passando o JWT no header:
```typescript
import { createClient } from '@supabase/supabase-js'

// Nao usar o client global — criar um dedicado com JWT customizado
function createTestClient(jwtPayload: Record<string, unknown>) {
  const token = buildJwt(jwtPayload) // ver abaixo
  return createClient(
    process.env.VITE_SUPABASE_URL!,
    process.env.VITE_SUPABASE_PUBLISHABLE_KEY!, // anon key
    {
      global: {
        headers: { Authorization: `Bearer ${token}` },
      },
    }
  )
}
```

Para RLS funcionar em testes de integracao, o JWT precisa ser assinado com o mesmo segredo
que o Supabase usa. Em ambiente de teste, usar o JWT_SIGNING_SECRET configurado como secret
da edge function (o mesmo que auth-token-exchange usa para gerar tokens).

Abordagem RECOMENDADA para os tests (mais simples, sem precisar assinar JWT):
Usar o Supabase service role key para executar SQL com `SET LOCAL request.jwt.claims`
dentro de uma transacao:

```sql

SET LOCAL request.jwt.claims = '{"org_id":"org_A","super_admin":"false"}';
SELECT COUNT(*) FROM documents;
ROLLBACK;
```

Isso permite testar RLS sem precisar de JWTs assinados.

transacao SQL com `SET LOCAL request.jwt.claims` para simular o JWT context, executa a query

IMPORTANTE: Se a funcao SQL `exec_with_claims` nao existir no banco, o executor deve criá-la
como parte do setup do test ou criar uma migration 021 que adiciona essa funcao helper:

```sql
-- Helper function para testes RLS (nao afeta producao)
CREATE OR REPLACE FUNCTION public.exec_with_claims(claims text, query_sql text)
RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
  result jsonb;
BEGIN
  PERFORM set_config('request.jwt.claims', claims, true);
  EXECUTE 'SELECT jsonb_agg(row_to_json(t)) FROM (' || query_sql || ') t' INTO result;
  RETURN COALESCE(result, '[]'::jsonb);
END;
$$;
```
