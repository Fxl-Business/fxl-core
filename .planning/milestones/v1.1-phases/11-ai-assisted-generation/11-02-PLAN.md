export type GenerationOptions = {
  vertical?: 'financeiro' | 'varejo' | 'servicos'

2. **`generateBlueprint(briefing: BriefingConfig, clientSlug: string, options?: GenerationOptions): BlueprintConfig`**

   Core logic:
   - If `options.vertical` is set, start from `structuredClone(VERTICAL_TEMPLATES[options.vertical])` as base
   - If no vertical, start with empty `{ slug: clientSlug, label: briefing.companyInfo.name, screens: [] }`
   - For each `briefing.modules[i]`:
     a. Call `findBestRecipe(mod.name, briefing.companyInfo.segment)` to get the best recipe
     b. Call `buildScreenFromRecipe(mod, recipe, briefing)` to construct a full BlueprintScreen
   - When using a template base, merge generated screens with template screens:
     - If a generated screen matches a template screen by id, overlay the generated data onto the template
     - If no match, append the generated screen
   - Set final config: `slug = clientSlug`, `label = briefing.companyInfo.name`, `schemaVersion = CURRENT_SCHEMA_VERSION`

3. **`buildScreenFromRecipe(mod: BriefingModule, recipe: ScreenRecipe, briefing: BriefingConfig): BlueprintScreen`** (internal helper)
   - Screen id: kebab-case from `mod.name` (e.g., 'DRE Gerencial' -> 'dre-gerencial')
   - Screen title: `mod.name`
   - periodType, hasCompareSwitch, filters: from recipe
   - For each `recipe.sections[j]`:
     a. Start with the recipe's `defaults` for that section
     b. Overlay module-specific data where applicable:
        - For kpi-grid: use `mod.kpis` to generate KPI items (label from kpi name, value as 'R$ 0')
        - For chart sections: use mod.name in title
        - For table sections: derive column labels from data context
     c. Ensure all required fields are present (merge with sensible defaults if recipe defaults are partial)

4. **Helper: `toKebabCase(str: string): string`** -- lowercase, replace spaces/special chars with hyphens

**Part A: CLI bridge script**

Create directory `tools/wireframe-builder/scripts/` and file `generate-blueprint.ts`:

```typescript
// Node.js CLI entry point -- NOT a Vite module
// Invoked via: npx tsx --env-file .env.local tools/wireframe-builder/scripts/generate-blueprint.ts <slug> [vertical]
```

1. **Supabase client creation** (standalone, NOT importing @/lib/supabase):
   - Read `VITE_SUPABASE_URL` and `VITE_SUPABASE_PUBLISHABLE_KEY` from `process.env`
   - Exit with clear error if either is missing
   - Create client: `createClient(url, key)`

2. **CLI argument parsing:**
   - `process.argv[2]` = clientSlug (required)
   - `process.argv[3]` = vertical (optional: 'financeiro' | 'varejo' | 'servicos')
   - `process.argv` check for `--force` flag anywhere in args
   - Exit with usage message if clientSlug missing

3. **Main flow:**
   a. Load briefing from `briefing_configs` table: `.from('briefing_configs').select('config').eq('client_slug', clientSlug).maybeSingle()`
   b. Validate briefing with `BriefingConfigSchema.parse(data.config)`
   c. Check if blueprint already exists: `.from('blueprint_configs').select('client_slug').eq('client_slug', clientSlug).maybeSingle()`
   d. If exists and no `--force`: log warning "Blueprint already exists for {slug}. Use --force to overwrite." and exit(1)
   e. Call `generateBlueprint(briefing, clientSlug, { vertical })` from `../lib/generation-engine`
   f. Validate output with `BlueprintConfigSchema.parse(blueprint)`
   g. Upsert to `blueprint_configs` table with `updated_by: 'claude:generation'`
   h. Log success: "Blueprint generated for {slug}: {N} screens"

4. **Error handling:**
   - Zod validation errors: log formatted error path and message
   - Supabase errors: log error message
   - All errors exit with process.exit(1)

Use ONLY relative imports for local modules (e.g., `../lib/generation-engine`, `../lib/blueprint-schema`). Do NOT use `@/` or `@tools/` aliases -- tsx does not resolve Vite aliases.
