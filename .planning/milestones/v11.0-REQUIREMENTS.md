# Requirements: Nexo — v11.0 Audit Logging

**Defined:** 2026-03-19
**Core Value:** Nexo e o hub central multi-tenant — cada empresa ve tudo sobre si mesma (modulos nativos + dados de apps externas) para que operadores e IA tenham contexto 360 graus.

## v11.0 Requirements

Requirements para sistema completo de audit logging. Cada requirement mapeia para uma fase do roadmap.

### Schema & Infraestrutura

- [x] **AUDIT-01**: Tabela `audit_logs` com schema enterprise (id, org_id, actor_id, actor_email, actor_type, action, resource_type, resource_id, resource_label, ip_address, user_agent, metadata JSONB, created_at) — imutavel via RLS (DENY UPDATE/DELETE)
- [x] **AUDIT-02**: Indexes compostos na mesma migration (org_id+created_at DESC, actor_id, action, resource_type, GIN em metadata)
- [x] **AUDIT-03**: DB triggers SECURITY DEFINER em tabelas criticas (tasks, tenant_modules) capturando INSERT/UPDATE/DELETE com before/after JSONB

### Captura de Eventos

- [x] **CAPT-01**: logAuditEvent() no service layer que nunca lanca excecao (falha reporta ao Sentry sem bloquear operacao principal)
- [x] **CAPT-02**: Instrumentacao dos Edge Functions admin-tenants e admin-users com logAuditEvent() para todas as acoes admin
- [x] **CAPT-03**: Captura de auth events (sign-in/sign-out) com IP e user-agent
- [x] **CAPT-04**: Impersonation context tagging — todos os logs durante sessao de impersonation carregam impersonator_id no metadata

### Query & API

- [x] **QUERY-01**: Edge function audit-logs com query paginada (LIMIT/OFFSET, max 100/request), filtros server-side, validacao super_admin JWT
- [x] **QUERY-02**: queryAuditLogs() no audit-service.ts como read path tipado

### UI Admin Panel

- [x] **UI-01**: Pagina /admin/audit-logs com tabela paginada (timestamp, actor, action, resource, org, IP) integrada no AdminSidebar
- [x] **UI-02**: Filtros composiveis: date range, action type (multi-select), actor (searchable), resource type, org (combobox)
- [x] **UI-03**: Detail drawer (Sheet) ao clicar numa row com todos os campos + before/after diff formatado para UPDATEs
- [x] **UI-04**: Export CSV dos logs filtrados (download imediato, native Blob API)

### Operacional

- [x] **OPS-01**: Retention policy configuravel em Platform Settings (default 90 dias) com pg_cron job de limpeza automatica

## Future Requirements

Deferred to v12+. Tracked but not in current roadmap.

### Per-Tenant View

- **TENANT-01**: Per-tenant audit log view scoped to org (para tenant admins, nao apenas super admin)
- **TENANT-02**: Inline audit history dentro de entity pages (ex: ultimas 5 mudancas de uma task)

### Advanced

- **ADV-01**: Anomaly detection alerts (actor com volume incomum de DELETEs)
- **ADV-02**: External log shipping para Sentry / Datadog / Papertrail
- **ADV-03**: Hash chaining para tamper-evident log chain (SOX-grade)
- **ADV-04**: DB triggers em blueprint_configs e briefing_configs
- **ADV-05**: Background export job para ranges grandes (>5000 rows) com download link

## Out of Scope

| Feature | Reason |
|---------|--------|
| SELECT-level logging | Volume explosion — reads podem ser 100x writes, destroi signal:noise |
| Real-time live-tail view | Sem valor de compliance, adiciona complexidade WebSocket |
| Full row snapshots para todas as tabelas | Risco PII, storage bloat — capturar apenas campos relevantes |
| Mutable audit logs com correcao | Destroi garantia de integridade — usar append-only correction events |
| Audit writes direto de React components | Re-render side effects, actor context indisponivel no browser |
| Raw HTTP request bodies nos logs | Contem PII, secrets, payloads grandes — viola principio de data minimization LGPD |
